# 바이브코딩으로 데이터과학 프로젝트 만들기 수업 노트

이 문서는 이 레포가 만들어진 과정을 학생들이 따라 배울 수 있게 정리한 노트입니다. 핵심은 한 번에 완성된 정답을 요구하는 것이 아니라, 질문을 구체화하고 결과를 확인하며 다음 요청을 더 정확하게 만드는 방식입니다.

## 1. 사용자가 먼저 제안한 방향

처음 방향은 "대전 교통사고 위험지역을 데이터과학 수업에서 볼 수 있는 웹앱으로 만들자"였습니다.

요청의 핵심은 다음과 같았습니다.

- 고등학생이 볼 수 있는 수준의 데이터과학 프로젝트여야 합니다.
- 대전 지역 교통사고 위험지역을 다루어야 합니다.
- 지도, 차트, 상관관계, 예측처럼 수업에서 설명할 요소가 필요합니다.
- GitHub에 올려 학생들이 열어볼 수 있어야 합니다.

이 단계에서 중요한 점은 주제가 명확했다는 것입니다. "교통사고"처럼 현실 데이터가 있고, "대전"처럼 지역 범위가 있으며, "고등학교 데이터과학 수업"처럼 사용자가 정해져 있으면 AI에게 훨씬 좋은 결과를 요청할 수 있습니다.

## 2. AI가 제안한 초기 구현

AI는 정적 웹앱 구조를 제안했습니다.

- `index.html`: 화면 구조
- `styles.css`: 디자인
- `app.js`: 데이터, 지도, 차트, 예측 모델
- `README.md`: 실행 방법과 수업 흐름
- `data/`: 샘플 CSV와 공공데이터 가공 파일

서버나 복잡한 프레임워크 없이 브라우저에서 바로 실행되도록 만든 이유는 수업 환경에서 설치 부담을 줄이기 위해서입니다.

## 3. 사용자가 확인하면서 제기한 문제

완성된 것처럼 보이는 앱도 실제로 확인하면 문제가 드러납니다. 이번 작업에서 사용자가 짚은 문제는 다음과 같았습니다.

- 수업용 샘플 데이터와 KOROAD 데이터 결과가 너무 비슷해 보였습니다.
- 추천 데이터셋도 같은 방식으로 정리해 넣으면 좋겠다는 요구가 있었습니다.
- 데이터셋마다 해석이 달라야 한다는 지적이 있었습니다.
- 지도 데이터가 이상해 보인다는 문제가 있었습니다.
- 새 데이터를 넣으면 해석이 어떻게 나오는지 설명이 필요했습니다.
- 해석본을 만들어 GPT 같은 도구에 넣어 해석하게 하는 방식도 제안되었습니다.

이 단계가 바이브코딩에서 가장 중요합니다. "이상하다"는 느낌을 그냥 넘기지 않고, 어떤 결과가 비슷한지, 어떤 설명이 부족한지, 어떤 사용 흐름이 어색한지 계속 확인해야 합니다.

## 4. AI가 추가로 제안하고 반영한 방향

사용자 피드백을 바탕으로 다음 방향을 반영했습니다.

- 데이터셋 선택지를 늘렸습니다.
- KOROAD 전체 사고다발지역, 보행자, 노인 보행자, 어린이보호구역 자료를 따로 선택할 수 있게 했습니다.
- 각 데이터셋마다 안내 문장과 해석 문장을 다르게 넣었습니다.
- 선택 지점의 해석에 중상, 어린이, 고령자, 야간, 교통량 같은 항목별 이유가 나오게 했습니다.
- 예측 결과 해설이 데이터셋의 성격을 반영하도록 바꾸었습니다.
- 지도에서는 대전 범위 좌표만 표시하게 했습니다.
- 실제 행정경계가 아닌 임의 구 경계 표시는 기본적으로 껐습니다.
- 전처리 스크립트를 추가해 원자료 JSON에서 수업용 CSV를 다시 만들 수 있게 했습니다.

여기서 배울 점은 "기능 추가"보다 "분석 목적에 맞게 해석을 바꾸는 것"입니다. 같은 모델이라도 보행자 사고 자료와 어린이보호구역 자료는 질문 자체가 다릅니다.

## 5. 데이터과학 프로젝트 요청을 잘하는 순서

학생들이 AI에게 요청할 때는 다음 순서를 권장합니다.

1. 주제와 대상을 정합니다.
   예: "대전 교통사고 데이터를 고등학교 데이터과학 수업용으로 분석하고 싶어."

2. 결과물을 정합니다.
   예: "지도, 차트, 상관관계, 간단한 예측 시나리오가 있는 웹앱으로 만들어줘."

3. 데이터 출처를 정합니다.
   예: "공공데이터포털과 KOROAD 자료를 사용할 수 있게 해줘."

4. 필요한 컬럼을 정합니다.
   예: "사고 건수, 사상자 수, 중상자 수, 좌표, 야간 비중, 어린이/고령자 변수를 포함해줘."

5. 먼저 샘플로 구현합니다.
   예: "실제 API 인증키가 없어도 수업 흐름을 볼 수 있는 샘플 데이터를 넣어줘."

6. 실제 데이터로 바꿉니다.
   예: "KOROAD 원자료를 앱 CSV 형식으로 전처리해서 선택할 수 있게 해줘."

7. 결과를 비교합니다.
   예: "샘플 데이터와 실제 데이터의 상위 위험 지점이 왜 다른지 설명해줘."

8. 해석과 한계를 붙입니다.
   예: "이 모델이 정책 결정용이 아니라 교육용이라는 점을 문서에 써줘."

9. 검증합니다.
   예: "좌표 범위, CSV 컬럼, 데이터셋별 상위 지점이 정상인지 확인해줘."

10. 기록하고 커밋합니다.
    예: "README와 수업 노트를 추가하고 커밋, 푸시해줘."

## 6. 좋은 요청 예시

나쁜 요청은 너무 추상적입니다.

```text
교통사고 데이터 분석 앱 만들어줘.
```

좋은 요청은 대상, 데이터, 기능, 해석 기준이 들어 있습니다.

```text
대전 지역 교통사고 공공데이터를 고등학교 데이터과학 수업용 정적 웹앱으로 만들어줘.
지도에는 사고다발지역 좌표를 표시하고, 차트는 지역별 사고 건수와 위험도, 시간대별 패턴, 속도 제한과 사고 규모를 보여줘.
위험도 모델은 교육용으로 만들고, 어떤 변수가 점수를 높였는지 학생이 설명할 수 있게 해석 문장도 넣어줘.
```

## 7. 결과를 확인할 때 볼 것

앱이 열리는지만 확인하면 부족합니다. 데이터과학 프로젝트에서는 다음을 확인해야 합니다.

- 데이터셋을 바꾸면 지점 수와 상위 위험 지점이 달라지는가
- 지도 마커가 실제 지역 근처에 찍히는가
- 좌표가 위도/경도 순서로 들어갔는가
- 필터를 바꾸면 차트와 예측 목록이 같이 바뀌는가
- 상관관계 값이 데이터가 적을 때 과장되어 보이지 않는가
- 위험도 점수가 왜 높거나 낮은지 설명할 수 있는가
- 샘플 데이터와 실제 데이터의 차이를 문서에 밝혔는가

## 8. GPT로 해석본을 만들 때의 프롬프트 구조

생성형 AI에게 데이터 해석을 맡길 때는 결과만 던지지 말고 맥락을 같이 줍니다.

```text
다음은 고등학교 데이터과학 수업용 교통사고 위험도 분석 결과입니다.
정책 결정용 확정 분석이 아니라 교육용 모델입니다.

데이터셋:
- 이름:
- 출처:
- 집계 기준:
- 행 수:

모델:
- 사고 건수, 중상 이상, 사상자 수, 어린이/고령자, 야간, 출퇴근, 제한속도, 교통량, 기상 취약도를 가중 합산했습니다.

상위 위험 지점:
1. 지점명 / 구 / 위험도 / 주요 변수
2. 지점명 / 구 / 위험도 / 주요 변수
3. 지점명 / 구 / 위험도 / 주요 변수

요청:
1. 왜 이 지점들이 높게 나왔는지 설명해줘.
2. 이 데이터셋의 한계를 알려줘.
3. 학생 토론 질문 3개를 만들어줘.
4. 과장된 결론을 피해서 조심스럽게 써줘.
```

이렇게 요청하면 AI가 "점수가 높으니 위험하다"에서 멈추지 않고, 변수와 한계를 함께 설명할 가능성이 높아집니다.

## 9. 이번 레포에서 학생이 배울 수 있는 것

이 프로젝트는 웹앱 제작 수업이면서 동시에 데이터과학 수업입니다.

- 데이터를 화면에 표시하는 방법을 배웁니다.
- 공공데이터를 수업용 데이터 구조로 전처리하는 방법을 배웁니다.
- 모델 점수가 변수 선택과 가중치에 따라 달라진다는 점을 배웁니다.
- 지도 시각화에서 좌표와 행정경계 검증이 중요하다는 점을 배웁니다.
- AI에게 한 번에 정답을 맡기는 것이 아니라, 결과를 보고 다시 요청하는 방식이 중요하다는 점을 배웁니다.

좋은 바이브코딩은 기분대로 만드는 것이 아니라, 원하는 결과를 말하고, 나온 결과를 확인하고, 이상한 점을 다시 질문하고, 그 기록을 남기는 과정입니다.
